@layout(post) @section(judul) Mengenal Mini SSG Zen @endsection @section(kelas) Mini SSG Zen @endsection @section(isi) @markdown Jadi, Mini SSG Zen ini adalah pengembangan dari tools yang sudah ada sebelumnya yaitu Mini SSG. Lantas, mengapa aku mengembangkan tools ini? Kenapa nggak [pull request](/post/panduan-dasar-menggunakan-git.html) aja ke repo utama (reponya Mini SSG)? Kalau PR (pull request) sih udah. Tapi kan, nggak mungkin ya aku request semua hal terkait kebutuhanku. Dan nggak mungkin juga semua request itu akan langsung diterima dan dimerge. Makanya, aku akhirnya fork itu repo kemudian aku kembangkan sendiri sesuai dengan kebutuhanku. Apa aja kebutuhanku? Pertama, aku kan pastinya pakai untuk mengembangkan aplikasi Android. Yap, aplikasi Android berbasis webview. Kenapa nggak pakai Java aja atau Flutter aja? Jawabannya simpel: Nggak bisa. Yang aku bisa kan berbasis webview. Jadi ya, itu aja deh yang aku buat. Kalau menggunakan Mini SSG aslinya, aku nggak bisa menggunakannya untuk mengembangkan aplikasi Android. Kok bisa? Soalnya kan, untuk partial komponen, dia meletakkannya di folder-folder yang diawali dengan underscore (`_components`, `_imports`, `_layouts`). Sedangkan, semua folder yang diawali dengan underscore akan diabaikan oleh Android Studio saat build. Jadinya, ketika aku ingin mengembangkan kembali aplikasi yang sudah aku buat, akan kesusahan. Kok bisa? Apakah karena aku nggak bisa mendapatkan source codenya? Yap. Aplikasi Android, yang berekstensi apk itu, kalau kita ekstrak, dia akan keluar isinya. Ya, berbentuk enkripsi sih. Kecuali untuk folder `assets`. Nah, di folder assets itulah aku meletakkan file-file HTMLnya. Kalau ada file atau folder yang berawalan dengan underscore, dia akan dihapus dari situ. Kan sayang. Susah mau modif lagi kalau sourcenya terhapus. Lalu, ada inkonsistensi dalam susunan filenya. Maksudku seperti ini: Kalau kita membuat `dev/pages/baca.html`, maka hasilnya nanti adalah `baca/index.html`. Padahal kan seharusnya itu ya `baca.html`. Makanya, di bagian ini juga aku baiki. Terus, di Mini SSG Zen ini, sudah aku sertakan dengan [Windi](/post/berbagai-cara-menggunakan-windi.html). Karena aku memang suka desain web dengan Windi sih. Kustomisasinya gampang banget eh. Windi itu kayak gini contohnya: ```css article { @apply p-5 bg-blue-500 border border-red-600 sm:(bg-blue-200 uppercase) } ``` Tapi ketika build, kode-kode Windi yang mengotori file HTML kita itu akan dihapus dan **semua kode Windi dijadikan satu dalam sebuah file `windi.css`** yang terletak di root. Lalu, kalau menggunakan Mini SSG aslinya, terkadang ketika kita dev, malah returnnya itu halaman kosong. Jadi kayak nggak tergabung gitu filenya dengan partial komponennya. Makanya, aku baiki juga dengan menambahkan parameter di Chokidar: ```diff chokidar.watch('./dev', { ignored: './dev/static/windi.css', + awaitWriteFinish: { + stabilityThreshold: 500 + } }).on('all', (event, path) => { runSSG() }); ``` Jadi, parameter `awaitWriteFinish` itu akan menunggu sampai semua file terolah, baru diwrite. Kemudian, kalau pengolahannya itu lama, akan langsung diabort dengan parameter `stabilityThreshold` yang bernilai 500. Aku belum paham sih 500 itu apa. Mungkin 500 milisekon atau 500 file. Nanti lah aku cari. Kemudian fitur yang aku sematkan pada Mini SSG Zen ini adalah fitur rendering Markdown. Untuk renderingnya, aku gunakan `markdown-it` lalu aku padu-padankan dengan Shiki untuk syntax highlightingnya. Sebagai bonusnya, aku sertakan juga auto generate Sitemap pas build. Asalkan disertakan file `mini.json` yang berisi seperti ini: ```json { "situs": "https://kucing.com" } ``` @endmarkdown @endsection
